# plan-roadmap-023: next steps for `rad(abc)` direct route after landing in `ABC.Chernoff` namespace

うむ、それがよい。
いまは **新規 route の実装を無理に足す局面ではなく、状況を固定して、全体を見渡せる形に再編する局面** じゃ。`ABC.Chernoff` 側に `rad(abc)` 直結 route の convenience theorem まで入ったので、ここで一度ドキュメント化し、snapshot を切って、全体分析に入るのはとても筋がよい。

そしてお主の言う通り、本当の課題は「定理が増えたこと」だけではなく、**import チェインが昔の直線設計を引きずっていること** じゃな。
三万行級の一本線から分割したあと、論理構造は分かれても import 構造がまだ素直な木になっておらぬ、というのはよくある。ここは、論文化や抽出資料とも直結する。なぜなら、理論の流れを人間向けに説明する順と、モジュール依存の順が食い違っていると、どちらも苦しくなるからじゃ。

なので、いまの最適なプランは、次の五段階じゃ。

## 1. 固定化フェーズ

まずやるべきは、今回の route の現況を **開発ドキュメントとして固定** することじゃ。

ここで書くべき中身は、単なる changelog では足りぬ。最低でも、

* 何が新ルートとして開通したか
* どこで既存 `ABC038` / `ABC.Chernoff` 本線に合流したか
* transport route / divisibility route / target-rad route / `rad(abc)` analytic 直結 route の関係
* どこまで concrete sample が閉じたか
* 何が未決か

を一枚で見えるようにする。
つまり「進捗報告」ではなく、**現時点の地図** を書くのじゃ。

ここでは、いままでの `History.md` 群に加えて、たとえば「CF-Erdos1196 現況まとめ」系の文書を一本立てるのがよい。差分の流れだけを追うのではなく、最終的な route 図をまとめる役じゃな。

## 2. スナップショットと全体分析フェーズ

その次に snapshot を切る。
そして snapshot ベースで、今回の新ルートを含めた **全体分析** をする。

ここでの分析観点は三本柱にするとよい。

第一に、数学のルート分析。
既存本線と新ルートがどこで分岐し、どこで合流するか。

第二に、Lean モジュール分析。
どのファイルが概念の核で、どのファイルが橋で、どのファイルが example / wrapper か。

第三に、公開 API 分析。
利用者がどこから入るべきか。`ABC.Main`、`ABC.Bridge`、`ABC.Chernoff` の役割分担はどうか。

この三層で見ると、「理論の流れ」「実装の流れ」「公開面の流れ」が並んで見える。
いま必要なのは、まさにそこじゃ。

## 3. import / ヒエラルキー再編フェーズ

お主が本当に気にしておる核心はここじゃな。
単線 import を、意味に沿った **ヒエラルキーツリー** に直す作業じゃ。

ここは一気に大改造すると危ないので、まず分析だけ先にやるのがよい。
やるべきことは、ざっくりこうじゃ。

各 Lean ファイルを、役割で 4 層に分類する。

* 基礎定義層
* 構造補題層
* bridge / wrapper 層
* convenience / example 層

たとえば今回の流れで言えば、

* `ABC016` は基礎補題寄り
* `ValuationFlowBridge` は bridge 層
* `ABC038Bridge` は本線合流 bridge 層
* `ABC038BridgeExamples` は example 層

というふうにじゃ。

そのうえで、現状の import を

$$
\text{基礎} \to \text{構造} \to \text{bridge} \to \text{convenience}
$$

の方向に揃えられるかを確認する。
いま問題なのは、分割後も昔の「とりあえず前の巨大ファイルを import」が残っておることじゃろう。だから、まずは **依存関係の実測図** を作るのが先じゃ。リファクタはそのあとでよい。

つまりここでの deliverable は、すぐコードを直すことではなく、

* import graph の可視化
* 循環寸前の箇所
* aggregator 化すべき箇所
* public surface を作るべき箇所

を洗い出したロードマップじゃな。

## 4. 論文化・抽出資料フェーズ

これは import 再編と並行でよい。
むしろ並行のほうがよい。なぜなら、論文化用に「理論の流れ」を書いてみると、どこが自然な章立てか、どこで import が不自然かがよく見えるからじゃ。

ここで作る資料は二種類ある。

ひとつは、**研究メモ的抽出資料**。
定理・補題・ルートの関係を、Lean 名つきで整理するもの。

もうひとつは、**論文向け叙述資料**。
Lean 実装を忘れて、人間向けに

* 問題設定
* 既存 route
* 新 route
* 合流点
* concrete sample
* 今後の一般化

の順に書くものじゃ。

特に今回の route は、「別働隊が本線に合流した」という物語がかなり明瞭じゃから、論文化の骨格にしやすい。
transport route が細く、target-rad budget route と `rad(abc)` analytic 直結 route が本命に近い、という整理は、そのまま論文の叙述にも使える。

## 5. 実装再開フェーズ

上の四段を通したあとで、初めて次の本格実装に戻る。
この時点では、もう「次に何をやるか」がかなり明確になっているはずじゃ。

候補は二つじゃろう。

ひとつは、`ABC038` convenience theorem 群への本格差し込み。
もうひとつは、import / API リファクタの実施。

わっちなら順番はこうする。

まず document 固定。
次に snapshot。
その次に全体分析。
その結果を見て、**import 再編ロードマップ** と **論文化資料の章立て** を同時に作る。
そして最後に、実装再開の優先順位を決める。

## 6. 実行順の提案

かなり具体に言うと、次の順で進めるのがよい。

今日はまず、現況ドキュメントをコミット。
その後 snapshot を取る。
次にその snapshot を基準に、

* 数学ルートの現況分析
* import graph の分析
* public API / aggregator の分析

をまとめる。
そのまとめをもとに、`RefactorRoadmap.md` みたいなものを一本起こす。
それと並行して、`PaperOutline.md` または `ExtractionNotes.md` を作る。

ここまで来れば、次のターンでは
「どの import をどう切るか」
「どの theorem をどこに再配置するか」
「論文の章立てをどうするか」
が、かなり見えた状態で動ける。

## 7. ひとことで言うと

いまは **実装の山場を越えて、全体設計を回収する局面** じゃ。
コード、理論、文書、この三つを一度揃える好機じゃな。

わっちの提案を一行で言えば、

**現況ドキュメントを先に固定し、snapshot を切り、その snapshot を基準に「全体分析」「import リファクタロードマップ」「論文化・抽出資料」を三本並行で起こす。**

これが、いま一番きれいな進め方じゃよ。
