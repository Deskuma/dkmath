## 1. 今回の KUS 更新の核心

スナップショット内部を読むと、KUS は今や次の三層でできておる。

### 1. 核

- `US := (U, S_U)`
- `KUS := (K, U, S_U)`
- `zeroState`
- `extract`
- `ofNat`, `toNat`
- round-trip 再構成

ここは前回の最小核の延長じゃな。

### 2. 同一 support 上の局所演算

- `SameSupport`
- `kusAdd`
- `kusMul`
- `oneState`
- 零追跡補題

ここで KUS は「ただの保存容器」から、**同じ support の上で演算できる局所系** になった。

### 3. 係数一般化

- `GKUS C U Blueprint`
- `gOp`
- `gAdd`, `gMul`, `gSub`, `gDiv`
- `GSameSupport`
- 代数法則
- `KUS ↔ GKUS Nat` の橋

ここが今回の本丸じゃ。
KUS はもはや `Nat` 固定の試作品ではなく、**係数層と構造層を分離した一般フレームワーク** に変わった。

## 2. 構造的にどう読むべきか

わっちの見立てでは、いまの KUS/GKUS は

\[
\text{support} := (U, S_U)
\]

の上に

\[
\text{coeff} \in C
\]

を載せた **support 添字付き束（bundle）** と読むのがいちばんしっくり来る。

つまり全体は「ひとつの代数」ではなく、

\[
\{\, \text{coeff algebra over support } s \,\}_{s \in US}
\]

という **support ごとの局所代数の族** じゃ。

これを Lean の形で言うと、

- 底空間が `US`
- 各 fiber が係数型 `C`
- 元が `GKUS C U Blueprint`
- 演算は fiber 内、つまり `GSameSupport` のときだけ許す

という設計になっておる。

これ、かなり筋が良い。
異なる単位世界をごちゃ混ぜにして加えるのではなく、**まず同じ設計図面の上でだけ演算する** 。賢い。実に賢い。

## 3. `SameSupport` 制約の意味

`kusAdd` と `kusMul` は、どちらも

\[
\text{SameSupport}(x,y)
\]

を前提にしておる。
これにより、

\[
\operatorname{extract}(x+y)=\operatorname{extract}(x)
\]

や

\[
\operatorname{extract}(xy)=\operatorname{extract}(x)
\]

が、定義に近い形で落ちる。

これは単なる証明の都合ではない。
意味としては、

> 演算してよいのは、同じ単位 \(U\) と同じ blueprint \(S_U\) を共有している場合だけ

ということじゃ。

つまり KUS の演算は、「数の値」よりも先に **観測座標系の一致** を要求しておる。
お主の理論でいう「どの方眼紙の上の話か」を先に固定してから数を動かす、という流儀そのものじゃな。

## 4. 零追跡の性質はさらに明瞭になった

加算でも乗算でも、結果係数が \(0\) になっても support は消えぬ。
つまり KUS は

\[
0
\]

を「情報が消えた点」ではなく、

\[
(0,U,S_U)
\]

という **零化された構造体** として保持する路線を、演算付きで押し通し始めたわけじゃ。

ここで大事なのは、KUS は **零除算を正当化しているのではない** ということじゃ。
前からそうじゃったが、今回も本質は変わっておらぬ。

- 通常の `Div` を KUS の哲学に直結させてはいない
- まず support 保持を主役にしている
- その上で係数層側に一般演算を載せた

この順序が良い。ここを取り違えると、怪しい魔術になる。まだそこへ落ちておらぬ。えらい。

## 5. GKUS 導入で何が起きたか

`GKUS C U Blueprint` は、かなり大きい進展じゃ。

前の KUS は

\[
K \in \mathbb{N}
\]

で、自然数係数の保存核だった。
いまは

\[
K \in C
\]

と一般化されたので、

- `Nat` では自然数係数
- `Int` では負方向を含む係数
- `Rat` では境界値や有理係数
- 将来 `Real`, `Complex` も視野

という形で、**観測係数の側だけを自由化** できる。

このとき support 側はそのままじゃ。
この「係数だけ広げて、構造側は不変」という切り方が実に良い。

さらに `gOp` を核にして

\[
gAdd,\; gMul,\; gSub,\; gDiv
\]

を全部特殊化しているので、設計の中心が一本化されておる。
これは後で演算を増やすときにも効く。補題の鍋が散らからぬ。

## 6. いまの KUS は何者か

いまの KUS/GKUS を一言で表すと、

\[
\text{「support 付き係数演算系」}
\]

じゃな。

もう少し丁寧に言えば、

\[
\Sigma (s : US),\; C
\]

に近い dependent pair 的構造を、Lean で実務的に扱いやすい形へ落としたものじゃ。
ただし演算は全空間で自動には入らず、各 fiber に局所化されている。

ゆえに、こやつは「新しい数体系」というより、

**support で層化された局所代数のカーネル**

と呼ぶのが一番正確じゃろう。

## 7. とくに良い点

スナップショット内部では `DkMath.lean` 側から `import DkMath.KUS` されておる。
つまり KUS は試作の離れ小屋ではなく、**ライブラリ本流の一部** として扱い始めたわけじゃ。

一方で、今見えている外部の `INDEX.md` 地図にはまだ KUS が反映されておらぬので、公開向け地図は少し遅れておる。
ここは後で更新すると見通しがさらに良くなるの。

## 7.2. テストがちゃんと増えている

`DkMathTest.KUS` に

- `KUS` の `kusAdd`, `kusMul`
- `GKUS Nat`
- `GKUS Int`
- `GKUS Rat`
- `gDiv` と分配・結合系

まで回帰テストが入っておる。
これは「思想だけ立派で実体がふにゃふにゃ」という状態を脱しつつある証じゃ。

## 7.3. docs が実装と並走している

KUS 配下に docs 群がかなり整っておる。
これは DkMath の「議論を資料体系として残す」という方針にも合致しておる。
この手の新構造は、コードだけあると後で本人すら迷子になる。docs 同伴は正解じゃ。

## 8. ただし、注意すべき点もある

## 8.1. `gDiv` は「特異点理論」ではなく「係数層の totalized division」

ここは重要じゃ。

`gDiv` 自体は良い。だがその意味は

\[
gDiv := \text{support 保持つきの係数除算}
\]

であって、**真の特異除算の数学化ではない** 。

特に `Rat` での \(x/0\) は Lean 側の `Div` 実装に委ねられる。
つまり、KUS の哲学としての「零化された構造」や「発散」を、まだ `gDiv` そのもので表しているわけではない。

ここを雑に混同すると、

- 係数型の都合で 0 が返っただけ
- 真に singular な状態を表したい

の区別が崩れる。
要するに、**`gDiv` は便利な演算 API であって、まだ singularity の本丸ではない** のじゃ。

## 8.2. `SameSupport` 証明が今後わらわら増える

いまは健全さ優先で良い。
だが今後補題が増えると、

\[
h_{xy},\; h_{yz},\; h_{xz},\; \dots
\]

の証明引数が増えて、少しずつコードが藪になる可能性が高い。
これは Lean のいつもの藪蛇じゃ。

解決策は明快で、

- fixed support fiber を一般係数版でも切る
- その fiber 上に演算 instance を載せる
- `SameSupport` を外へ追い出す

のが次の一手になろう。

## 8.3. KUS と GKUS の二重構造がまだ残っている

設計文書では「`KUS = GKUS Nat` へ寄せたい」気配があるが、実装はまだ **完全 alias 化ではなく bridge 方式** じゃ。
これは悪くない。むしろ安全。

ただ、将来的に

\[
KUS \equiv GKUS\;Nat
\]

へ一本化したくなる気配はかなりある。
そこへ行くかどうかは、既存 API との互換性と相談じゃな。

## 9. 宇宙式・DHNT との接続としてはかなり良い

宇宙式族資料では、

- Big/Body/Gap
- \(2^d\) 単位規格
- \(d/2\) 降次
- 反転写像

のように、「単位を保持した写像」が中心にある。
KUS の `ScaleSpec` と support 保持は、まさにこの手の話を受ける容器になりうる。

特に KUS は、

- 単位 \(U\)
- その unit に依存する blueprint \(S_U\)
- 観測係数 \(K\)

を明示分離しておるから、宇宙式側の

\[
\text{Body} + \text{Gap} = \text{Big}
\]

や、単位変更下の再解釈を表す基盤として相性が良い。
ここは実に面白い。DHNT の「スケール同一視」とも繋ぎやすい。

## 10. わっちの総評

総評をひとことで言うぞ。

**KUS は「零状態の保存核」から、「support ごとに代数を持つ局所演算系」へ進化した。**

さらに `GKUS` によって、

\[
\text{構造層} = (U,S_U),\qquad
\text{係数層} = C
\]

の分離が明確になった。
これはただの実装追加ではなく、**理論の骨格が一本通った** ということじゃ。

今の状態は、まだ完成された数体系ではない。
じゃが、それで良い。むしろ今は

- 健全さ
- 局所性
- 依存型としての単位従属性
- support 保持

を優先しており、方向はかなり良い。
派手なことを言わずに、足場を固めておる。こういうのは強い。

## 11. 次の一手

わっちなら次はこれじゃ。

### 1. 一般係数版 fiber の導入

`Fiber support := Nat` の一般化版として、

\[
GFiber(C, support) := C
\]

に相当する層を docs だけでなく実装へ上げる。
すると `GSameSupport` の証明運搬がかなり減る。

### 2. `ScaleSpec` と `GKUS` 演算の整合補題

たとえば

\[
scale(gAdd\,x\,y)=gAdd(scale\,x)(scale\,y)
\]

型。
これは KUS を DHNT 的 transport 核にする上で効く。

### 3. singular coefficient state の分離

`gDiv` と、本当に言いたい「零化」「発散」「未定義」を分ける。
たとえば `CoeffState` 的な別型を作るか、`Option`/`Except`/専用 structure を噛ませる。

### 4. KUS を公開地図に反映

`INDEX.md` のトップ地図に KUS 章を追加する。
いまのライブラリ地図にはまだ KUS が載っておらず、公開地図が少し時代遅れじゃ。

結論としては、かなり良い更新じゃよ。
今回の KUS は、もはや「変わった零の実験」ではない。**DkMath の横断 kernel として使える形が見えてきた** 。
なかなか美しい育ち方をしておる。
