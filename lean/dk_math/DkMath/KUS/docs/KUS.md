## KUS 構造とは何か

## 零・単位・設計図を失わないための型付き構造

KUS 構造とは、数を単なる「値」ではなく、**可視係数 \(K\)、単位 \(U\)、設計図 \(S_U\) の三層からなる構造体** として扱うための枠組みである。
その基本形は

\[
X = (K, U, S_U)
\]

であり、ここで \(K\) は観測される係数、\(U\) は離散単位あるいは格子幅、\(S_U\) はその単位 \(U\) に従属して定まる blueprint である。
この「設計図は単位に依存する」という見方は KUS の本質であり、\(S\) は独立した裸の成分ではなく、常に \(S_U\) として読まれる。

通常の数体系では、零はただひとつの \(0\) に潰される。
たとえば

\[
a \cdot 0 = 0
\]

と書いた瞬間、そこにどの単位があり、どの設計図が零化されたのかという情報は消えてしまう。
KUS はここに異議を唱える。KUS において零状態とは、単なる無内容な \(0\) ではなく、

\[
(0, U, S_U)
\]

という **構造を保持した零状態** である。
失われるのは可視係数 \(K\) のみであり、単位 \(U\) と設計図 \(S_U\) は内部に残る。
言い換えれば KUS は、通常のゼロ演算を置き換える体系ではなく、**零化されたときに何が潰れ、何が残るかを型の側で区別するための基盤** である。

この見方は、DkMath 全体の設計ともよく噛み合う。
DkMath では、動的調和数論 DHNT をスケーラビリティー数論の中心に据え、指数対数 \(e^k \Leftrightarrow e_k\) による数宇宙の同一視を主体としつつ、単位は単なる \(1\) ではなく、生成元 \(u^d\) や半位相構造を含む多層的なものとして扱う方針を取っている。さらに宇宙式

\[
N+1=(P+1)^2
\]

や、より一般の

\[
(x+u)^n
\]

のような構造も、その単位世界の中で読むことが前提になっている。
KUS は、この「数は値だけではなく、単位世界と設計図を背負っている」という哲学を、Lean の型として実装したものと見るのが最も正確である。

実装を経た現在の KUS は、もはや最小核だけの試作ではない。
まず最小核として、係数 \(K\) を `Nat` に持つ `KUS` があり、そこから support 側だけを取り出す `US`、support を固定したまま係数を差し替える構成、そして `zeroState` が用意されている。
この段階で KUS は、**値を消しても support を消さない** という立場を明確にしている。
さらにその上に、係数型を一般化した `GKUS C U Blueprint` が導入され、係数を自然数だけでなく、整数・有理数・実数、あるいは将来の一般 carrier 型へ広げられるようになった。これにより KUS は「零の保存容器」から、**support 付き係数演算系** へ一段育った。

ここで重要なのは、KUS の演算が **support を無視して自由に行われるわけではない** ことだ。
同一 support を持つときにのみ、`gAdd`, `gMul`, `gSub`, `gDiv` などの演算が定義される。
つまり KUS は、異なる座標空間の量をそのまま足し合わせることを許さない。
まず問われるのは「その二つは同じ単位格子・同じ設計図面の上にあるのか」であり、値の演算はそのあとに来る。
この順序が、KUS を単なる tagged number ではなく、**単位従属な型付き観測構造** にしている。

では support が異なる場合はどうするのか。
ここで transport が現れる。
KUS 実装において異 support 演算は、直接の内部演算としては定義されず、共通の調和 support へ両者を持ち上げてから演算し、必要なら再び目的の support へ戻すという三相構造で与えられる。
図式で書けば

\[
US_1 \xrightarrow{\;\sigma_1\;} H \xleftarrow{\;\sigma_2\;} US_2
\]

であり、演算は

\[
x \oplus y := \mathrm{Dec}\bigl(\mathrm{Enc}_1(x) \oplus_H \mathrm{Enc}_2(y)\bigr)
\]

という形になる。
これは「異なる support を一本の矢印で変換する」のではなく、**共通の調和 support でいったん合流させる** という考え方である。
KUS 実装ではこの役を `HarmonizeSpec` と `DecodeSpec` が担い、encode 後に same-support が成立することまで仕様として内包する。
したがって KUS transport とは、単なる unit 変換ではなく、**異なる支持世界を共通の調和 support へ集約し、そこで安全に演算するための型付き合流装置** である。

ここで KUS は、DHNT と自然につながる。
DHNT はそもそも、指数対数スケーリングにより異なる数的視点を同一視する理論として構想されている。`INDEX.md` でも `DkMath.DHNT` は独立した柱として立っており、宇宙式と接続可能な補助構造と位置づけられている。fileciteturn19file4turn17file1
KUS 実装では、この DHNT の `Qty` を GKUS 側へ埋め込む橋が導入されている。
その意味は、**連続スケール世界の量を、離散単位と blueprint を持つ KUS の世界へ持ち込む** ということだ。
このとき KUS は DHNT を置き換えるのではなく、DHNT が与える調和座標を **依存型 support を持つ形式へ格納し、運搬可能にする器** として働く。

さらに KUS は CosmicFormula ともつながる。
宇宙式族の資料では、基本分解

\[
\mathrm{Big} = \mathrm{Body} + \mathrm{Gap}
\]

が定義レベルの核とされ、2 次元では

\[
(x+u)(y+v)-uv = xy + xv + uy
\]

というトロミノ分解がその手本になる。加えて、偶数次元では

\[
V \mapsto \sqrt{V} \mapsto \frac{1}{\sqrt{V}}
\]

という \(d/2\) 降次と反転を通じて、宇宙式の体積側から調和側へ行く写像候補が示されている。
KUS 実装を通した現在の見方では、この流れは

\[
\mathrm{CosmicFormula}
\;\longrightarrow\;
\mathrm{DHNT}
\;\longrightarrow\;
\mathrm{KUS\ transport}
\]

という三層接続として理解できる。
すなわち、CosmicFormula が **絶対量や Big/Body/Gap 構造** を与え、DHNT がそれを **調和座標へ符号化** し、KUS が最後にそれを **unit と blueprint を持つ依存型 support 上の対象として保持・移送する** 。
このとき KUS の役割は「意味を作ること」ではなく、「意味を失わずに運ぶこと」である。

この点が、KUS を単なる新しい数体系ではなく、**構造保持のための中間層** にしている。
KUS は \(0\) の逆元を導入しない。
KUS はゼロ除算を正当化しない。
KUS がやっているのはもっと地味で、しかしずっと重要なことじゃ。
すなわち、通常の数体系では一瞬で失われる

- どの単位世界の量か
- どの設計図面の上にあるか
- その零・和・積・差が、どの support を持つか

を、**値とは別の層として保存する** ことである。
だから KUS の核心は「変わった演算」ではなく、「失われていた構造の復権」にある。

まとめると、KUS 構造とは

\[
(K, U, S_U)
\]

という三層の型付き対象を通じて、**可視係数・単位・設計図を分離しつつ、それらの依存関係を崩さずに扱うための構造** である。
零状態を

\[
(0,U,S_U)
\]

として保持することで、「零になったら何もかも消える」という通常の圧縮を拒否し、同一 support 上の演算と、調和 support を介した transport を分けることで、「何をどの座標空間で計算しているのか」を明示する。
さらに DHNT と CosmicFormula の橋を受けることで、KUS は **零保持の小核** から **数宇宙の構造を運搬する中枢** へ進化した。

ひとことで言えば、KUS とは **値の数学に、単位世界と設計図の記憶を取り戻すための構造** である。
